# 政务主动推荐智能体整体流程设计

## 1. 目标定位

本智能体面向政务办公场景，基于用户上下文、页面上下文、业务数据和知识库内容，为用户主动推荐政策文件、服务事项、待办任务、通知公告和办理建议。

核心目标是让用户在政务办公过程中尽量做到：

- 不用找：主动聚合与当前场景相关的信息。
- 不用查：结合用户身份和页面场景理解用户可能需要什么。
- 不用等：优先提示临期、重要、强相关内容。
- 不出错：推荐结果需要有来源、有理由、有操作建议。
- 不越权：默认只使用用户当前可见或接口返回的数据。

## 2. MVP 设计原则

MVP 阶段采用“框架完整、节点轻量”的方式建设。

也就是说，工作流节点先完整保留，但每个节点先做最小能力，先跑通一条完整链路，再逐个优化。

## 3. 总体流程

```text
开始节点
→ 固定信息获取与上下文汇总
→ 用户场景理解
→ 推荐任务规划
→ 多源数据召回
→ 候选内容标准化
→ 数据范围确认
→ 推荐排序
→ 推荐理由生成
→ 最终回复生成
→ 反馈记录
→ 结束节点
```

## 4. 节点职责概览

| 节点 | 核心职责 | MVP 实现方式 |
|---|---|---|
| 开始节点 | 接收用户输入和调用方传入的上下文 | 用户输入 + 3 个上下文参数 |
| 固定信息获取与上下文汇总 | 获取智能体可自行获取的数据，并汇总上下文 | Code 节点 + HTTP 节点 |
| 用户场景理解 | 理解用户、页面、业务和当前需求 | LLM 节点 |
| 推荐任务规划 | 决定本次推荐目标、召回范围和排序策略 | LLM 节点 |
| 多源数据召回 | 获取待办、政策、事项、通知等候选内容 | HTTP / 知识库 |
| 候选内容标准化 | 将不同来源内容统一格式 | Code 节点 |
| 数据范围确认 | 确认只使用用户可见数据 | MVP 透传，接口侧保证 |
| 推荐排序 | 选择最值得推荐的 3-5 条内容 | Code 或 LLM |
| 推荐理由生成 | 为每条推荐生成理由和操作建议 | LLM 节点 |
| 最终回复生成 | 生成用户可见回答 | LLM 节点 |
| 反馈记录 | 记录推荐结果和用户行为 | HTTP，可先占位 |

## 5. 开始节点参数

调用方只需要传入最小上下文，不需要把所有业务数据都传进来。

如果 Dify 已有默认用户输入，则开始节点额外配置 3 个上下文参数。

```text
business_scene
recommendation_context
custom_context
```

如果 Dify 没有默认用户输入，则补充 `user_input`。

```text
user_input
business_scene
recommendation_context
custom_context
```

参数含义：

- `user_input`：用户当前输入。
- `business_scene`：当前业务场景，包含时间、地点、页面、模块、业务场景和当前对象。
- `recommendation_context`：本次推荐业务、推荐目标、推荐类型、推荐重点和推荐数量。
- `custom_context`：调用方自定义上下文，可放用户 ID、租户、当前对象、mock 数据或额外约束。

MVP 固定业务场景：

```text
政务工作台首页 - 普通办公人员日常办公辅助推荐
```

## 6. 数据来源边界

调用方负责传入页面最清楚的信息。

智能体负责自行获取稳定信息，例如：

- 用户画像。
- 用户待办。
- 用户通知。
- 常用事项。
- 政策知识库检索结果。
- 服务事项知识库检索结果。
- 反馈日志。

## 7. 推荐输出要求

最终输出面向用户，不暴露内部 JSON、节点名称、意图识别结果或排序规则。

推荐结果建议包含：

- 推荐标题。
- 推荐类型。
- 推荐理由。
- 建议操作。
- 来源。
- 可选跳转入口。

## 8. 后续优化方向

MVP 跑通后，按节点逐步优化：

- 推荐不准：优化用户场景理解和任务规划。
- 数据不全：优化多源召回。
- 顺序不合理：优化推荐排序。
- 理由不可信：优化推荐理由生成。
- 用户体验弱：优化最终回复生成。
