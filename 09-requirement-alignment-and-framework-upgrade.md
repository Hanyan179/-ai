# 原始需求对齐与框架升级说明

## 1. 结论

当前设计没有偏离原始需求主线，但仍偏向“推荐生成链路 MVP”。

原始需求描述的是一个完整的“辅助推荐智能体产品能力体系”，不仅包括推荐生成，还包括：

- 角色管理。
- 对话流程管理。
- 提示词管理。
- 知识工程。
- Tools 工具体系。
- 流程编排、异常恢复和效果优化。

因此，当前框架可以作为 MVP 主链路，但需要在框架层补齐若干节点和能力占位。

## 2. 当前设计已覆盖的部分

| 原始需求 | 当前设计对应节点 | 覆盖程度 |
|---|---|---|
| 数据收集与整合 | 开始节点、固定信息获取与上下文汇总 | 已覆盖主干 |
| 上下文场景数据 | `business_scene`、`recommendation_context`、`custom_context` | 已覆盖 |
| 用户画像构建 | 用户场景理解节点 | 部分覆盖 |
| 场景分析与感知 | 用户场景理解节点 | 已覆盖主干 |
| 潜在需求挖掘 | 推荐任务规划节点 | 部分覆盖 |
| 个性化推荐生成 | 推荐排序、理由生成、最终回复生成 | 已覆盖主干 |
| 个性化推荐列表 | 最终回复生成节点 | 已覆盖 |
| 推荐结果排序 | 推荐排序节点 | 已覆盖 |
| 反馈优化 | 反馈记录节点 | 占位覆盖 |

## 3. 当前设计不足的部分

### 3.1 角色管理不足

原始需求包含：

- 辅助推荐人格设定。
- 角色切换。
- 权限配置。
- 权限管控规则。

当前设计只在输出风格中隐含了“政务办公辅助推荐专员”，但没有形成独立节点。

应补充：

```text
角色与权限配置加载节点
```

MVP 可先固定角色：

```text
政务办公辅助推荐专员
```

后续再支持角色切换。

### 3.2 行为数据和长短期记忆不足

原始需求强调：

- 浏览历史。
- 操作记录。
- 搜索关键词。
- 长短期记忆。
- 用户偏好学习。

当前设计只写了用户画像和反馈记录，没有把行为数据、记忆数据作为固定数据源明确写入。

应补充到固定信息获取阶段：

```text
用户行为数据获取
短期会话记忆获取
长期偏好记忆获取
用户画像更新
```

### 3.3 知识图谱关联不足

原始需求强调：

- 政策文件与业务指南关联。
- 服务事项与数据报告关联。
- 待办任务与协同事项关联。
- 通过知识图谱挖掘潜在需求。

当前设计主要使用知识库检索，尚未明确“知识图谱关联节点”。

应补充：

```text
知识图谱关联与潜在需求挖掘节点
```

MVP 可先用规则或知识库关联替代知识图谱。

### 3.4 推荐意图识别和非推荐需求处理不足

原始需求要求：

- 识别是否为推荐需求。
- 排除非推荐类需求。
- 对系统操作咨询、公文写作咨询等进行引导。

当前设计把这一能力放入“用户场景理解”和“推荐任务规划”里，但没有单独显式标出。

应补充：

```text
推荐需求判定节点
```

注意该节点不是普通闲聊分类，而是判断：

```text
当前需求是否属于辅助推荐服务范畴
```

### 3.5 场景模板库不足

原始需求包含：

- 文书创作场景模板。
- 政策查询场景模板。
- 年度总结创作场景指令。
- 政策申报场景指令。

当前设计只有页面推荐提示，没有独立模板库。

应补充：

```text
推荐场景模板匹配节点
```

MVP 可先内置少量模板。

### 3.6 异常、超时、中断处理不足

原始需求包含：

- 推荐超时处理。
- 推荐中断恢复。
- 检索失败处理。
- 行为数据缺失处理。
- 知识图谱关联失效处理。
- 非法输入处理。

当前设计只保留了反馈记录，没有完整异常链路。

应补充：

```text
异常检测与降级推荐节点
流程状态记录节点
```

MVP 可先采用简单降级策略：

- 行为数据缺失：使用页面场景推荐。
- 知识库无结果：使用服务入口或人工咨询入口。
- 接口失败：使用已获取数据生成基础推荐。

### 3.7 知识工程能力表达不足

原始需求包含：

- NaiveRAG。
- InstructRAG。
- 结构化框架生成。
- 内容适配补全。
- 合规性前置校验。
- 个性化记忆。
- 检索失败处理。

当前设计已有“多源召回、标准化、排序”，但没有明确区分检索方式和补全策略。

应在知识工程层补充：

```text
基础检索节点
指令式检索节点
推荐结构化框架生成节点
推荐内容适配补全节点
合规性前置校验节点
```

### 3.8 Tools 工具体系未显式映射

原始需求列出了完整 Tools：

- 推荐数据源融合适配 Tool。
- 推荐策略动态调整 Tool。
- 政务数据联动 Tool。
- 推荐需求解析 Tool。
- 推荐内容匹配 Tool。
- 知识图谱关联 Tool。
- 推荐合规校验 Tool。
- 用户偏好学习 Tool。
- 个性化推荐生成 Tool。
- 推荐结果优化 Tool。
- 推荐效果分析 Tool。
- 异常推荐修复 Tool。
- 工具调度与异常处理 Tool。

当前设计在节点上覆盖了一部分，但没有形成 Tool 对照表。

应在后续增加：

```text
Tools 映射设计文档
```

## 4. 对齐原始需求后的完整框架

建议将总流程升级为：

```text
开始节点
→ 角色与权限配置加载
→ 固定数据获取与上下文汇总
→ 数据预处理
→ 推荐需求判定
→ 用户画像构建与记忆读取
→ 场景分析与模板匹配
→ 知识图谱关联与潜在需求挖掘
→ 推荐任务规划
→ 多源检索
   ├─ 基础检索 NaiveRAG
   └─ 指令式检索 InstructRAG
→ 候选内容标准化
→ 推荐内容适配补全
→ 合规性前置校验
→ 推荐排序
→ 个性化推荐生成
→ 推荐结果优化
→ 最终回复生成/结果推送
→ 反馈记录与记忆更新
→ 推荐效果分析
→ 结束节点
```

异常和超时处理作为全局兜底能力：

```text
任一数据源失败
→ 异常检测与降级推荐
→ 基础推荐结果
→ 反馈记录
```

## 5. MVP 阶段保留的最小可运行版本

虽然完整框架需要上述节点，但 MVP 不需要把每个节点都做复杂。

建议 MVP 按以下方式实现：

| 框架节点 | MVP 做法 |
|---|---|
| 角色与权限配置加载 | 固定为“政务办公辅助推荐专员”，权限由接口侧保证 |
| 固定数据获取与上下文汇总 | 获取用户画像、行为数据、待办、通知、知识库结果；为空也继续 |
| 推荐需求判定 | LLM 输出是否属于推荐服务范畴 |
| 用户画像构建与记忆读取 | 先读取已有用户上下文和最近行为，不训练模型 |
| 场景分析与模板匹配 | 基于页面场景和推荐提示匹配少量内置模板 |
| 知识图谱关联与潜在需求挖掘 | 先用规则或关联字段替代真实图谱 |
| 多源检索 | HTTP + 知识库，四类数据都可召回 |
| 合规性前置校验 | 先做“只使用接口返回可见数据”校验 |
| 推荐排序 | 简单规则排序 |
| 个性化推荐生成 | LLM 生成推荐列表 |
| 反馈记录与记忆更新 | 先记录日志，不做复杂学习 |
| 异常恢复 | 接口失败时输出基础推荐和失败说明 |

## 6. 开始节点参数是否仍然成立

成立。

但口径调整为“用户输入 + 3 个上下文参数”。

如果 Dify 已经有默认用户输入，则开始节点额外配置：

```text
business_scene
recommendation_context
custom_context
```

如果没有默认用户输入，则补充：

```text
user_input
```

需要明确：这些参数只是“调用方传入的最小上下文”，不是智能体全部输入。

智能体完整输入应由两部分组成：

```text
调用方页面上下文
+ 智能体自行获取的系统数据
```

智能体自行获取的数据至少包括：

```text
用户画像
用户行为数据
短期会话记忆
长期偏好记忆
待办任务
通知公告
服务事项
政策文件
知识图谱关联结果
角色配置
权限配置
推荐模板
反馈记录
```

## 7. 最终判断

当前设计：

```text
适合作为 MVP 推荐链路
```

但如果对照原始需求：

```text
还需要升级为“辅助推荐智能体完整能力框架”
```

所以不是方向偏了，而是目前只覆盖了“推荐生成主链路”，还没有把原始需求中的角色、记忆、知识图谱、模板、异常、工具和效果优化完整显式化。
